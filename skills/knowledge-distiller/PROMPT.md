# 生成 Skill 的提示词模板

> 使用说明：将下方「」内的占位内容替换为你自己的情况，整段复制发给 AI，它会生成一套完整的 Skill 文件包。

---

## 提示词正文

```
请帮我生成一套完整的 Skill 文件包，用于在支持 Skill 的 AI 编辑器（如 Trae、Cursor、Claude）中调用。

---

## 一、这个 Skill 是干什么的

「用一两句话说清楚这个 Skill 的核心目的。
例：通过 AI 主动追问，帮助用户把脑中模糊的隐性知识说清楚、整理好、存下来。」

适用场景：「列举 3-5 个典型的使用场景。
例：
- 我有某个领域的经验，但说不清楚
- 我有一个模糊的想法，想搞清楚逻辑
- 我想把某段经历整理成可复用的方法论」

不适用场景：「说清楚这个 Skill 不该干什么，帮助 AI 划清边界。
例：不是用来搜索资料的，不是用来生成文章的」

---

## 二、AI 扮演的角色

「明确告诉 AI 它是谁、它的任务是什么、它不该做什么。
例：你是一位专业知识挖掘者和采访者。你的任务不是回答问题，而是通过追问帮助用户把脑中模糊的知识说清楚。你不应该主动给出结论或建议，除非用户的观点有明显错误。」

---

## 三、核心行为规则

「列出这个 Skill 运行时 AI 必须遵守的规则，写得越具体越好。
以下是一个参考框架，按实际需求增删：」

1. 单点原则：每次回复只做一件事（只问一个问题 / 只给一条反馈）
2. 先承接后推进：先简短回应用户刚说的内容，再进入下一步
3. 定期整理：每进行「N」轮对话，主动整理一次，生成阶段性摘要请用户确认
4. 首要任务是「说清楚」还是「说正确」：
   - 如果首要任务是帮用户把自己的想法表达清楚：遇到错误也要先追问，不要急着纠正
   - 如果需要纠错：遇到明显事实错误时温和指出，遇到不完整的观点主动补充，纠正后继续追问不打断节奏
5. 错误记录：用户说错或说不完整的内容，在最终输出里单独标注（例：标注"已纠正"或"已补充"）
6. 拒绝空泛：用户回答过于抽象时，必须追问具体案例
7. 破冰引导：用户不知道聊什么时，主动引导（问最近发生了什么 / 遇到了什么问题）
8. 触发整理：当用户说「「整理一下」/ 「结束」/ 「停止」」时，按 output_template.md 生成最终笔记

「根据你的 Skill 补充其他规则：」
9. 「自定义规则...」

---

## 四、启动流程

「描述这个 Skill 被调用后，AI 应该按什么顺序执行。流程越清晰，执行越稳定。
例：」

1. 加载 @reference/modes.md，向用户展示可选的对话模式
2. 请用户选择模式（说明可以组合使用）
3. 用户确认后，问：「你设计的开场问题」
4. 正式开始

---

## 五、对话模式（如果有多种工作方式）

「如果这个 Skill 有多种工作模式，在这里列举。
每种模式说明：名称 / 适用场景 / AI 的行为方式 / 一段示例对话。
如果模式只有一种，这部分可以省略，直接合并进核心规则。」

模式 1：「模式名称」
- 适用场景：「...」
- AI 行为：「...」
- 示例：「写一小段对话示例」

模式 2：「...」

---

## 六、输出格式

「描述这个 Skill 结束时应该生成什么样的输出。
如果需要固定格式的 Markdown 文档，在这里列出所有板块的名称和说明。
例：」

最终笔记包含以下板块：
- 对话日期 / 领域 / 本次主题 / 使用模式
- 核心观点列表（含被纠正后的版本）
- 支撑案例
- 判断标准
- 答不上来的问题（认知边界）
- 认知纠偏记录（原始说法 → 纠正后）
- 可延伸写作的方向

「根据你的 Skill 调整板块，删掉不需要的，加入特有的」

---

## 七、持久化存储（如果需要跨对话积累数据）

「如果这个 Skill 需要记住用户的历史状态，在这里说明。
如果不需要跨对话积累，这部分可以省略。
例：」

- 每个「领域 / 项目 / 用户」单独建立一个档案文件，存入 domains/ 目录
- 档案内容包括：背景信息、当前状态、历次摘要、待解决问题
- 每次对话结束后，将本次核心内容追加至对应档案，待解决问题同步更新

---

## 八、文件结构要求

请按以下结构生成文件，**不要把所有内容塞进一个文件**：

```

「skill名称」/
├── SKILL.md                      # 入口文件，角色定义 + 核心规则 + 启动流程，保持精简
├── reference/
│   ├── modes.md                  # 对话模式详细说明（如有）
│   ├── output_template.md        # 最终输出的 Markdown 模板
│   └── domain_template.md        # 持久化档案的空白模板（如有）
└── domains/                      # 空目录，运行时按需新建档案文件

```
**模块化要求：**
- SKILL.md 只写角色、规则、启动流程，保持精简，引用子文件用 `@reference/文件名` 格式
- modes.md 写每种模式的详细说明和示例对话，SKILL.md 里只列模式名称
- output_template.md 只写输出的空白模板，不写规则
- domain_template.md 只写档案的空白模板，文件头加使用说明注释
- 不要在任何一个文件里重复写其他文件已有的内容

---

## 九、其他要求

- SKILL.md 的 frontmatter 里加上中英文触发词（triggers），覆盖用户可能说的各种表达
- domain_template.md 里加上状态字段的判断标准说明
- 每个文件逐一输出完整内容，用 markdown 代码块包裹，文件名标注在代码块上方
- 输出顺序：SKILL.md → reference/modes.md → reference/output_template.md → reference/domain_template.md

---

## 十、自检要求

生成完所有文件后，在输出结尾附上一份自检报告，逐项确认以下内容是否符合要求：

- [ ] SKILL.md 里引用子文件的方式是 `@reference/文件名`，不是超链接
- [ ] SKILL.md 里有明确的"何时更新档案"指令（如果有持久化需求）
- [ ] 启动流程是编号步骤，不是一段话
- [ ] modes.md 里每种模式都有示例对话
- [ ] domain_template.md 里有使用说明注释，状态字段有判断标准
- [ ] 没有任何一个文件把其他文件的内容重复写了一遍

如有未满足的项，直接在对应条目后说明原因或给出修复建议。
```

## 使用技巧

**填写建议**

填得越具体，生成质量越高。尤其是以下三个地方：

- **核心行为规则**：不要只写"帮我追问"，要写清楚"什么时候追问、怎么追问、遇到什么情况做什么"。规则越细，AI 执行越稳定。
- **示例对话**：每种模式至少给一段对话示例，哪怕只有 3 轮，AI 会照着这个风格生成后续内容。
- **输出格式**：把你希望最终笔记长什么样画出来，AI 会按照你的预期生成 output_template.md。

**生成后必检清单**

已内置在提示词第十节，AI 生成完文件后会自动逐项核查并报告结果，无需手动检查。
