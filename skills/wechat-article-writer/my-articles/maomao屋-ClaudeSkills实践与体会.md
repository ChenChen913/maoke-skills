# 2、名词汇总与解释

### **元提示词（****Meta** **Prompt）**

这是一个“**一招鲜，吃遍天**”的绝招，元提示词就是**提示词的提示词**，因为在日常使用 AI 过程中，总会遇到一些陌生领域的问题，用户对于这种问题可能连怎么提问都不会，比如：我想让 AI 帮我设计一个前端页面，但是我不懂前端的知识，我也只能简单的说一句“帮我设计一个前端网页，好看一点······”

但是你可以用元提示词来引出你的问题，可以先这样说：

> 我想让 AI 帮我做一个美观的前端页面，你帮我写一个向 AI 提问的提示词吧

然后 AI 就会给你一个很详细且结构化的提示词，然后直接将该提示词给 AI 就可

### 元技能/元 skill（Meta Skill）

这个和元提示词是一个道理，就是**生成 Skill 用的 Skill**，官方已经给出了一个 skill-creator，当你想创建一个 skill 的时候，CC 就会自动调用这个 skill-creator 帮你完善好一个 skill。

### 标准作业程序（SOP，Standard Operating Procedure）

这个就是大家常说的 SOP，也是对 Skills 的一种比喻，还有另外一种比喻是：Skills 就像是给新员工做入职培训。

这两种比喻都形象地说明了：Skill 就是一种限制、一个说明书、一种规则······而 CC 在执行特定任务时就按照 Skill 中的具体说明来运行

### 渐进式披露（Progressive Disclosure）

这是 Skill 在 CC 中的触发原理，就是**层层递进、按需读取**！

因为 SKILL.md 文件主要是由三层信息组成（**元数据**必须加载，**指令**触发时加载，**资源和代码**按需加载），每一层信息的读取顺序是固定的，其实这和优先级有点像，每一个文件中优先级最高的信息被优先读取，尽管文件中不同层级信息的优先级不同，在实际执行任务时，所有层级的信息并不一定都被会读取，只有优先级最高的那层信息是一定会被读取的

### 垂直 Agent

可以这样理解：Claude Code + 特定领域的 Skills = 垂直 Agent

本来大模型是什么都懂的通用辅助 AI，但是针对每一个特定领域懂得不多，然后我把某个小领域的可复用的技术技能抽象出来（就是上面说的 SOP），告诉 AI 当遇到这类问题时具体应该怎么做，这时 Agent 就迅速获得这个特定领域的专业能力

而 ClaudeCode Skills 为构建垂直 Agent 提供了低成本的革命性方法！

# 4、Skills 上手实践

## 4.1 安装 skill-creator

前面也提到了元 skill 的概念，Anthropic 官方提供的这个 skill-creator 就是用来创建新 skill 用的，至于安装方式无非就两种

（ 官方给的 GitHub Skills 库： https://github.com/anthropics/skills) 

1. 自然语言安装：告诉 CC 这个 skill 的 Github 地址，让它自己把文件配置好；
2. 手动下载：还是去 Github 上自己把项目包下载下来，然后将 skill-creator 文件夹复制到相应的目录下（比如我会把文件夹复制到这：C:\Users\11853\.claude\skills) 

CC 启动后就可以使用/skills 命令测试一下是否配置成功，如果配置成功，skill-creator 的名称就会出现在列表中，如果说某个 skill 你觉得不好不想用了，直接删除对应的文件夹就行，比如：我不想用 skill-creator 了，就去 C:\Users\11853\.claude\skills 路径下把 skill-creator 删掉即可

然后我在创建手动创建其他新的 skill 的时候发现两个问题

1. 和 CC 说创建新 skill 时，它根本就不会就调用 skill-creator！！！
2. 自己创建 skill 时，skill 相关联的文件中会出现 Emoji 或者一些奇怪的符号，因为在对 skill 进行测试的时候经常会提醒因为 Emoji 图标测试失败！

于是我借助 Claude Sonnet-4.5 帮我把这两个问题修复了一下，就是在 skill-creator 的 SKILL.md 文件中增加两条新的约束（在每创建一个新的 skill 时都会让其在 YAML 处加上 triggers 以增加 skill 的触发条件，另外不要在 skill 关联的文件中使用 Emoji 或者其他奇怪的符号）

## 4.2 创建 skill-evaluator

有了一个生成 skill 的 skill，好像还少点什么，对！**还缺一个 skill 审核员，就是需要再创建一个审核 skill 的 skill**，（不排除在一些 SKills Marketplace 里已经有类似的 skill）之所以想到这个主要还是担心 skill 的安全问题，在这篇文章中讲过【】

> 一个创建 skill 的 skill，一个审核 skill 的 skill，简直就是日常开发的左膀右臂啊！

我先是大概构思了一下 skill 审核员要干的工作，于是给 Claude Sonnet-4.5 抛了这样一个问题：

```Markdown
最近在ClaudeCode中使用skills比较方便，但是大量的skill容易有一些质量比较差的skill滥竽充数，甚至会引发安全问题，下面是我打算通过skill-creator来创建一个skills审核员，我写的提示词很粗糙，你帮我修改补充一下，最好是重写
<content>
skills审核员，一是看依赖附加捆绑文件，二是看有没有诱导AI进行危险性操作，三是有没有执行任务时会不会因为中文等出现编码问题，四是有没有对环境有特殊要求、主要适合什么系统；五是看一下skill的markdown部分写地是否规整条例和有逻辑；六是看这个skill有没有什么明显的局限性，或者有没有什么可以扩展的地方
</content>
```

然后它就给我了一个很专业的 skill 审核员的 md 文件，然后启动 CC 让其创建一个相关的 skill，创建时参照这个 md 文件即可

这是它给我写得审核 skill 的提示词，还是很专业的：

~~~Markdown
# Skills 质量审核员

你是 ClaudeCode 的综合技能质量审核员。你的职责是系统化地评估 skill 文件，提供详细、可执行的反馈，以确保质量、安全性和可用性。

## 审核框架

对以下维度进行全面分析：

### 1. 依赖与文件捆绑分析
- **外部依赖**：识别所需的所有外部库、包或工具
- **捆绑文件**：检查是否包含附加文件、模板或配置文件
- **依赖风险**：评估依赖项中的潜在安全漏洞
- **版本规范**：验证依赖版本是否固定或灵活
- **安装要求**：评估安装步骤是否清晰且最小化

### 2. 安全与风险评估
- **危险操作**：识别可能导致以下后果的命令：
  - 未经明确确认删除或修改系统文件
  - 从外部来源执行任意代码
  - 访问敏感系统资源（密码、密钥、凭证）
  - 向不受信任的端点发送网络请求
  - 修改系统配置或权限
- **提示词注入风险**：检查 skill 是否可能被操纵以绕过安全准则
- **数据泄露**：评估意外数据泄漏或敏感信息记录的可能性
- **用户确认**：验证破坏性操作是否需要明确的用户同意

### 3. 编码与国际化兼容性
- **字符编码**：测试 UTF-8 兼容性，特别是：
  - 中文、日文、韩文（CJK）字符
  - 特殊符号和表情符号
  - 包含非 ASCII 字符的文件路径
- **区域设置处理**：检查 skill 是否正确处理不同的系统区域设置
- **字符串处理**：识别以下方面的潜在编码问题：
  - 文件 I/O 操作
  - 命令行参数
  - 输出格式化
- **错误消息**：验证错误消息无论字符编码如何都清晰明了

### 4. 环境与平台兼容性
- **操作系统**：明确支持的平台（Linux/macOS/Windows）
- **系统要求**：记录任何特殊要求：
  - 最低软件版本
  - 所需的系统工具或实用程序
  - 需要的环境变量
  - 文件系统权限
- **跨平台问题**：识别特定于平台的代码路径或限制
- **优雅降级**：检查 skill 是否能优雅地处理缺失的依赖

### 5. 文档质量评估
评估 markdown 文档的以下方面：
- **结构与组织**：
  - 清晰的标题和用途说明
  - 合理的章节层次结构
  - 正确使用标题级别（H1、H2、H3）
- **完整性**：
  - 安装/设置说明
  - 包含预期输出的使用示例
  - 参数描述
  - 错误处理指导
- **清晰度**：
  - 简洁、无术语的语言
  - 适当的分步说明
  - 带解释的代码示例
- **格式规范**：
  - 一致的 markdown 语法
  - 带语言标签的正确代码块格式
  - 列表和表格格式正确

### 6. 功能分析与改进机会
- **当前局限性**：
  - 未处理的边缘情况
  - 已知的 bug 或变通方法
  - 性能限制
  - 扩展性问题
- **潜在增强**：
  - 能显著增加价值的缺失功能
  - 更好的错误处理机会
  - 可能的优化
  - 与其他工具的集成机会
- **最佳实践差距**：skill 偏离编码最佳实践的地方

## 输出格式

以以下结构化格式提供审核报告：

### 审核摘要
[一段概述：skill 用途、总体质量评级（优秀/良好/一般/差），以及关键问题（如有）]

### 详细发现

#### 优势亮点
- [列出关键优势]

#### 发现的问题
**[类别名称]** - 严重程度：[严重/高/中/低]
- **问题**：[描述]
- **影响**：[后果说明]
- **建议**：[具体修复或缓解措施]

#### 改进建议
- [可执行的改进想法]

### 最终建议
- [ ] **批准** - 可直接使用
- [ ] **有条件批准** - 可用但建议改进
- [ ] **需要修订** - 必须解决重大问题
- [ ] **拒绝** - 存在严重的安全或质量问题

---

## 审核指导原则

1. **客观公正**：基于可观察的事实进行评估，而非假设
2. **安全优先**：立即标记任何安全问题
3. **建设性反馈**：提供具体、可操作的建议
4. **考虑上下文**：根据 skill 声明的目的和范围进行评估
5. **逻辑推演**：遍历代码逻辑以识别潜在故障点
6. **用户视角**：考虑最终用户体验和常见用例

当不确定安全问题时，宁可谨慎标记以供审查。

## 审核检查清单

使用此清单确保全面审核：

- [ ] 检查所有依赖项和外部文件
- [ ] 扫描危险命令和操作
- [ ] 测试中文路径和文件名兼容性
- [ ] 验证跨平台兼容性说明
- [ ] 评估文档完整性和清晰度
- [ ] 识别功能局限性和改进空间
- [ ] 为每个问题提供具体修复建议
- [ ] 给出明确的批准/拒绝建议
```
~~~

```Markdown
如果是中间，你需要对事情已经发生的情况做一个分析，有哪些好的地方和不好的地方，好的应该怎么保持，不好的应该怎么改进或者避免，并根据现状对下一步甚至是很长的一段时间的后半阶段做一个计划

如果是结束，你需要好好的总结整件事的过程，有哪些好的地方和不好的地方，好的应该怎么保持，不好的应该怎么改进或者避免，哪些地方是因为性格、能力、沟通等地方出了问题，从这些事里面你能学到什么，有什么感想，下一次再遇到类似的事情就得有一个大体的规划

中间还有很多东西要细节化！比如听谁的意见，如何知道自己的想法是不是任性和不合逻辑的······

</content>
```

# 5、推荐安装的 Skills

看到很多人推荐的几个：

（1）**skill-creator**：辅助创建 skill 的 skill

仓库地址：https://github.com/anthropics/skills/tree/main/skills

（2）**Superpowers**：一个高度结构化的技能库，让 Claude Code 扮演更“自主开发者”的角色，配备设计思维、测试驱动开发（TDD）、规划和执行等流程化的工程工作流，提升编程效率和输出质量。

仓库地址：https://github.com/obra/superpowers

（3）**NotebookLM skill**：能在 Claude Code 里面直接和 NotebookLM 对话，并上传 PDF 直接到 NotebookLM

仓库地址：https://github.com/PleasePrompto/notebooklm-skill

（4）**obsidian-skills**：提供 Obsidian 相关语法和知识库支持的一套技能，例如 Obsidian 风格的 Markdown、Bases 数据库语法，以及 JSON Canvas 可视化内容等

仓库地址：https://github.com/kepano/obsidian-skills

（5）**Planning-with-files**：这是 Manus 的核心工作方法！让 Claude Code 在复杂任务中不是只依赖上下文窗口，而是以 Markdown 文件作为长期计划和执行记录，从而更好管理多步任务和错误追踪。

仓库地址：https://github.com/OthmanAdi/planning-with-files

有篇文章还单独对它的 SKILL.md 文件做了详细的讲解：【AI 街溜子】拆解一个 3k Star 的 Skills：它到底写了什么？顺便还能学习 manus 的上下文管理

（6）**frontend-design**：生成有鲜明设计风格、生产级质量的前端界面代码，避免 AI 生成千篇一律的无特色 UI

仓库地址：https://github.com/anthropics/skills/tree/main/skills/frontend-design
