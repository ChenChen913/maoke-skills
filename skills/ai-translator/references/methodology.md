# 翻译方法论详细参考

本文档包含翻译提示词的 8 大核心要素详细说明，供翻译过程中参考。

---

## 要素 1：角色定位（Role Assignment）

为 AI 分配专业角色身份，使输出符合专业标准。

**构建方式**：
- 明确专业领域：`你是一位专业的{领域}技术文档翻译专家`
- 强调语言能力：`精通英文和简体中文，了解两种语言的技术表达习惯`
- 突出专业知识：`具有深厚的{领域}专业知识和行业经验`

**模板**：
```
你是一位资深的{领域}技术文档翻译专家，专门从事{类型}的英译中翻译工作。
你精通英文和简体中文，熟悉两种语言在技术文档中的表达习惯和行业规范。
你拥有{背景}，能准确理解和转换复杂的技术概念。
```

## 要素 2：任务说明（Task Instruction）

清晰具体地说明翻译任务要求。

**必须明确**：
- 源语言和目标语言
- 文档类型（API 文档、论文、用户手册等）
- 翻译目的（内部文档、公开发布、学术交流等）
- 翻译倾向（直译/意译/平衡）

**示例**：
```
请将以下英文技术文档翻译为简体中文。
这是一份面向开发者的 API 文档，将用于产品的官方中文文档网站。
翻译时需要保持技术准确性，同时确保中文表达流畅自然。
```

## 要素 3：术语管理（Terminology Management）

技术翻译中最关键的要素，确保术语准确性和一致性。

### 术语表分级结构

**核心术语（必须遵守）**：固定翻译，不允许变化
```
Machine Learning = 机器学习
Neural Network = 神经网络
API = API（保持不翻译）
```

**领域术语（根据上下文）**：
```
Model = 模型（AI/ML 场景）| 模式（设计模式场景）
Pipeline = 数据流水线（数据工程）| 构建流水线（CI/CD）
```

**可选术语（建议使用）**：
```
Training Data = 训练数据
Hyperparameter = 超参数
```

### 术语处理原则

1. 明确哪些术语保留英文（API, HTTP, JSON 等）
2. 首次出现中英对照：`机器学习（Machine Learning）`
3. 后续出现仅用中文：`机器学习`
4. 保持英文的术语首次也要注释：`Transformer（一种注意力机制架构）`
5. 专有名词（公司名、产品名）保持原文

### 多义词上下文校准

遇到多义词时，按以下优先级判断：
1. 参考术语表中的上下文规则
2. 参考文档的技术领域
3. 参考该词在前后文中的语境
4. 不确定时标注备选译法供用户确认

## 要素 4：语气和风格（Tone and Style）

### 正式程度选择

- **正式技术文档**：专业、严谨
- **用户友好文档**：清晰、易懂，避免过度技术化
- **营销材料**：吸引人、有说服力
- **学术论文**：客观、严谨、规范

### 避免翻译腔（核心要求）

**被动语态转主动**：
- "It is recommended that..." → "建议..." 或 "请..."
- "It should be noted that..." → "需要注意..."
- "It can be seen that..." → "可以看出..."

**长难句拆分**：
- 包含 3 个以上从句的长句，拆分为 2-3 个短句
- 保持逻辑清晰，每句表达一个核心意思

**品牌语调一致性**：
- 使用主动语态
- 避免冗长从句
- 保持简洁明了
- 对用户友好

## 要素 5：文化本地化（Cultural Localization）

### 度量衡与格式

- 日期：MM/DD/YYYY → YYYY 年 MM 月 DD 日
- 数字：10,000 → 10000 或 1 万
- 百分比保留：95% → 95%

### 文化适应

- 避免直译英文习语，使用中文对应表达
- 调整不符合中文阅读习惯的表达方式

### 排版规范

- 使用中文标点符号（，。、；：""（））
- 中英文之间添加空格
- 中文与数字之间添加空格
- 数字与单位之间添加空格（百分号除外）

---

## 要素 6：格式保持（Format Preservation）

### 必须保留的格式

- Markdown 标记：标题、加粗、斜体、列表、代码块
- HTML 标签（如有）
- 表格结构
- 代码示例（仅翻译注释）
- URL 和链接（保持 URL 不变，翻译链接文字）
- 图片引用

### 代码处理规则

1. 所有代码保持英文不翻译
2. 代码注释翻译为中文（单行、多行、文档字符串）
3. 函数名、变量名、类名保持英文
4. JSDoc、PyDoc 参数说明翻译
5. 保持代码缩进和格式不变

---

## 要素 7：上下文信息（Context Provision）

为 AI 提供充分的背景信息：

- 文档在整个体系中的位置
- 预期读者的技术水平
- 使用场景
- 产品/项目基本介绍
- 前置知识要求
- 领域特定约定

---

## 要素 8：质量标准（Quality Criteria）

### 四维质量要求

**准确性**：术语正确、数字一致、不遗漏不添加

**流畅性**：中文自然、符合语言习惯、无翻译腔

**一致性**：术语全文统一、语气风格统一

**完整性**：不遗漏任何内容、保留所有细节

### 自查清单

翻译完成后检查：
- [ ] 术语是否正确且一致
- [ ] 中文表达是否自然流畅
- [ ] 格式是否完整保留
- [ ] 是否遗漏或添加信息
- [ ] 数字、数据是否准确
- [ ] 标点符号是否规范

## 高级技巧

### 多步骤提示

```
步骤 1：直译，保持术语准确性
步骤 2：优化中文表达，使其流畅自然
步骤 3：检查术语一致、格式完整
```

### 示例引导（Few-Shot）

提供 2-3 个高质量翻译示例，让 AI 学习目标风格。

### 反译验证

适用于高风险文档（产品公告、法律条款、安全警告）：
1. 翻译 EN→CN
2. 反译 CN→EN
3. 对比原文与反译文
4. 修正偏差

### 约束条件

```
1. 译文长度不超过原文的 1.3 倍
2. 必须保留所有技术细节
3. 禁止用意译简化复杂概念
4. 不得省略警告或注意事项
5. 保持原文语气（警告用强语气，建议用缓和语气）
```
