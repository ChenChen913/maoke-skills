# Tutorial Writer

编写准确、有时效性、结构清晰的技术教程。**默认输出为 Markdown 文件。**

---

## 工作流总览

```
Step 0: 创建任务规划文件（_task_plan.md）
Step 1: 判断教程类型，选择模板
Step 2: 检查可用的搜索能力
Step 3: 网络调研
Step 4: 生成 4 个版本文件，综合选优
Step 5: 两轮审核
Step 6: 输出最终 Markdown 文件
Step 7: 清理所有临时文件
```

---

## Step 0：创建任务规划文件

开始任何工作之前，先在工作目录创建 `_task_plan.md`：

```markdown
# 教程任务规划

## 基本信息
- 教程主题：[填写用户需求]
- 判断类型：[填写]
- 选用模板：[填写]
- 创建时间：[填写]

## 任务进度
- [ ] Step 1：判断类型 & 选择模板
- [ ] Step 2：确认搜索工具
- [ ] Step 3：网络调研完成
- [ ] Step 4a：生成 version-1.md
- [ ] Step 4b：生成 version-2.md
- [ ] Step 4c：生成 version-3.md
- [ ] Step 4d：生成 version-4.md
- [ ] Step 4e：综合选优，生成 draft-final.md
- [ ] Step 5a：第一轮审核（正确性）
- [ ] Step 5b：第二轮审核（逻辑完整性）
- [ ] Step 6：输出最终文件
- [ ] Step 7：清理临时文件

## 调研摘要
（Step 3 完成后填写关键发现）

## 审核记录
（Step 5 发现的问题及修正简记）
```

每完成一个步骤，立即将 `_task_plan.md` 中对应的 `[ ]` 改为 `[x]`，并补充相关记录。这个文件是上下文超长时的恢复锚点。

---

## Step 1：判断教程类型

教程来源不限于固定分类，根据用户描述灵活判断。常见举例（不限于此）：

- 用户给了 GitHub 链接、仓库名，或将项目下载到本地 → 项目类
- 涉及具体工具或框架（Git、Docker、Claude Code、Trae、VS Code 等）→ 工具类
- 涉及安装、配置、部署某个软件（如安装配置 OpenClaw）→ 安装配置类
- 涉及概念/理论（计算机网络、设计模式、CAP 定理等）→ 概念类
- 对比多个工具帮助读者选型 → 对比选型类
- 介绍某种工作流程或最佳实践 → 工作流类
- 其他来源 → 选用最接近的模板，结构可灵活调整

完成后在 `_task_plan.md` 中填写类型和模板，标记 Step 1 为 `[x]`。

---

## Step 2：检查可用的搜索能力

1. 查看 available_skills 列表，确认是否有专门的搜索 Skill
2. 查看用户记忆文件或配置文件，确认是否有搜索工具偏好
3. 有指定工具则优先使用，否则使用默认 web_search

标记 Step 2 为 `[x]`。

---

## Step 3：网络调研

必须执行，不可跳过。

**时效性核查**：以今天实际日期为基准，搜索最新稳定版本号、近期重大变更、是否有废弃功能或更推荐的替代方案。

**准确性调研**（按优先级）：
1. 官方文档或官方 README
2. 官方 CHANGELOG 或 Release Notes
3. 近期技术文章或社区讨论
4. 用户提供了本地项目文件时，优先阅读源码和文档目录

调研完成后，在 `_task_plan.md` 的"调研摘要"部分记录关键发现，标记 Step 3 为 `[x]`。

---

## Step 4：生成 4 个版本文件，综合选优

### 4a–4d. 在工作目录生成四个 Markdown 文件

每个文件都是一份完整的教程草稿，基于相同的调研材料，但切入角度不同：

- `version-1.md`：以"背景与动机"为切入点，先建立认知，再进入操作
- `version-2.md`：以"最快跑通"为目标，极简前置，直接进入实操
- `version-3.md`：以"核心概念拆解"为主线，先建完整认知框架再动手
- `version-4.md`：以"真实场景驱动"为主线，从具体问题出发引出解决方案

每生成一个文件，在 `_task_plan.md` 中标记对应条目为 `[x]`。

### 4e. 综合选优

读取四个版本文件，按以下标准评判：逻辑链清晰度、示例可运行性、表达简洁度、结构合理性。

可以选出最优版本，也可以综合多个版本的优点合并为更好的草稿。
将结果写入 `draft-final.md`，标记 Step 4e 为 `[x]`。

---

## Step 5：两轮审核

### 第一轮：正确性审核
- 所有命令和代码是否与最新稳定版本匹配
- API 名称、参数名是否与官方文档一致
- 是否使用了已废弃的写法
- 示例代码是否可独立运行

发现问题直接修正，在 `_task_plan.md` 审核记录中简记，标记 Step 5a 为 `[x]`。

### 第二轮：逻辑与完整性审核
- 各章节顺序是否符合认知规律
- 前置条件是否完整
- 章节之间是否有断层
- 概念类：类比是否准确，不会产生误导

同样直接修正，记录，标记 Step 5b 为 `[x]`。

若发现重大问题（如核心功能已废弃），需在最终输出时告知用户。

---

## Step 6：输出最终文件

将审核完成的 `draft-final.md` 另存为最终教程文件，命名格式：

```
[主题名]-tutorial.md
```

例如：`claude-code-tutorial.md`、`docker-tutorial.md`

在文件末尾追加两个固定章节：

**本教程的范围说明**
- 适合的读者层次和覆盖范围
- 未能覆盖的内容及建议查阅的资源类型

**延伸学习资源**（通过搜索获取真实链接）
- 官方文档（直链 + 版本说明）
- B 站视频（高播放量、近期更新的中文讲解）
- YouTube（官方频道或知名技术博主）
- 书籍/课程（书名、作者、出版年份）

将最终文件呈现给用户，标记 Step 6 为 `[x]`。

---

## Step 7：清理临时文件

删除以下全部临时文件：
`_task_plan.md`、`version-1.md`、`version-2.md`、`version-3.md`、`version-4.md`、`draft-final.md`

只保留最终的 `[主题名]-tutorial.md`。标记 Step 7 为 `[x]`（在删除 `_task_plan.md` 前完成标记）。

---

## 写作规范

- 不使用 emoji 和装饰性图标
- 不以碎片化条目堆砌代替连贯表达
- 代码块必须注明语言（```bash、```python 等）
- 专有名词保持原文（不翻译 Docker、Git 等工具名）
- 语言风格：清晰、直接，像一位有经验的同事在讲解

---

## 思维模型（概念类教程专用）

当教程需要解释抽象原理时，考虑是否有合适的思维模型辅助讲解。详细说明见 `mental-models/` 目录。

**使用前必须先告知用户：**
1. 准备使用哪个思维模型
2. 该模型适合解决什么类型的问题
3. 该模型的基本结构和思路
4. 等待用户确认后，再正式开始写作

可选模型速查表：

| 模型 | 适用场景 | 详情 |
|------|----------|------|
| 第一性原理 / MECE / 抽象阶梯 / 组块化 | 拆解与分析 | `mental-models/analysis.md` |
| SWOT / 权衡三角 / 对比学习 | 对比与权衡 | `mental-models/comparison.md` |
| 系统思维 / 因果链 / 冰山模型 | 系统与因果 | `mental-models/systems.md` |
| 费曼技巧 / 80/20 / 黄金圈 / 决策树 / 类比推理 | 简化与传授 | `mental-models/simplification.md` |
