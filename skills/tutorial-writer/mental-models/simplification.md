# 思维模型：简化与传授类

---

## 费曼技巧（Feynman Technique）

**适合场景**：作为整篇教程的基础写作原则，确保内容不依赖术语堆砌，对目标读者真正可读。

**基本思路**：
诺贝尔物理学奖得主理查德·费曼的学习方法：如果你不能用简单的语言向外行解释一个概念，说明你自己还没有真正理解它。将这个原则应用到教程写作上，意味着：写每一个概念时，先想象你在向一个聪明但不熟悉这个领域的人解释它，然后再写。

**写作步骤**：
1. 写完一段解释后，检查：其中有哪些词是"行话"或需要前置知识才能理解的？
2. 对每一个这样的词，要么在首次出现时解释它，要么用更通俗的替换词
3. 检查：能否用一个类比或日常场景来说明这个概念的核心？
4. 去掉所有"可以不写的句子"——只有帮助理解的内容才留下

**适用提示**：费曼技巧适合在写完初稿后，作为"自审工具"来检查每一段文字。不是要把教程写得幼稚，而是确保清晰。

## 80/20 法则（Pareto Principle）

**适合场景**：教程篇幅有限，或主题太大无法全覆盖时，用于决定讲解内容的优先级。

**基本思路**：
大约 20% 的核心知识点，能覆盖 80% 的实际使用场景。教程不需要面面俱到，找到那 20% 并深入讲透，比浅尝 100% 更有价值。

**写作步骤**：
1. 列出该主题所有可以讲的知识点
2. 对每个知识点评估：它在实际使用中出现的频率如何？
3. 选出频率最高的 20%（通常是 3–5 个核心知识点）作为主体内容
4. 其余 80% 在"延伸学习资源"中给出指引，不在正文中展开
5. 在"范围说明"中清楚告知读者：本教程聚焦哪些内容，哪些内容有意略过

## 黄金圈（Golden Circle: Why → How → What）

**适合场景**：读者需要被"说服"先理解价值，才会有动力学操作时。适合面向开发者推广新工具、新框架、新工作方式的教程。

**基本思路**：
Simon Sinek 提出的框架：大多数人讲解事物的顺序是 What → How → Why，但最有说服力的顺序是反过来——先讲 Why（为什么这个东西值得你花时间学），再讲 How（它是怎么工作的），最后讲 What（具体是什么、怎么用）。

**写作步骤**：
1. **Why**：这个工具/技术解决了什么真实的痛点？没有它，开发者会遇到什么麻烦？
2. **How**：它是用什么机制或思路来解决这个问题的？（概述，不需要细节）
3. **What**：具体的安装步骤、API、命令行用法

**适用提示**：黄金圈不适合"查手册"类教程（读者已经知道为什么要用，只需要知道怎么用）。更适合推广性质的入门教程。

## 决策树（Decision Tree）

**适合场景**：教程的核心内容是帮助读者根据不同情况做出正确选择时。

**基本思路**：
将复杂决策分解为一系列"是/否"或"选 A/选 B"的判断节点，形成树状结构。读者沿着与自己情况匹配的路径走，最终到达适合自己的答案。

**写作步骤**：
1. 列出影响决策的所有关键维度（如：是否需要持久化？是否有高并发要求？团队规模？）
2. 确定维度的优先级（最重要的判断放在树的第一层）
3. 用文字或表格呈现决策路径
4. 在每个终点给出具体的方案和理由

**常见应用**：
- "如何选择数据库"类教程
- "选择哪种部署方式"类教程
- "什么情况下用 X，什么情况下用 Y"类教程

## 类比推理（Analogical Reasoning）

**适合场景**：概念完全陌生，读者没有任何前置知识可以依附时，用来建立初始的直觉认知。

**基本思路**：
将陌生的技术概念映射到读者已经熟悉的日常领域。好的类比能让读者在几秒内建立"大概是什么感觉"的直觉，为后续精确学习打下基础。

**写作步骤**：
1. 找到该概念最核心的特征（不超过 2–3 个）
2. 在日常生活中寻找具有相同特征的事物
3. 用类比解释核心特征，同时明确指出类比"在哪里不准确"（所有类比都有边界）
4. 在建立了初步直觉之后，给出精确的技术定义

**注意**：类比是入门工具，不是精确描述。一定要在类比之后给出准确的技术解释，并指出类比的局限，避免读者把类比当成真实机制来理解。

**常见示例**：
- DNS 类比于"电话簿"（优：说明了名称到地址的映射；限：没有体现分布式、缓存、TTL）
- 线程类比于"同一个厨房里的多个厨师"（优：共享资源、并发操作；限：没有体现线程调度）
- Git commit 类比于"游戏存档"（优：状态快照、可以回退；限：没有体现有向无环图结构）
