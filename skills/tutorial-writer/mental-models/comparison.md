# 思维模型：对比与权衡类

---

## SWOT 分析

**适合场景**：需要系统性地介绍某个技术方案，并与其他方案做横向比较，帮助读者做选型判断时。

**基本思路**：
从四个维度分析一个方案：
- Strengths（优势）：该方案在哪些方面表现突出
- Weaknesses（劣势）：该方案的固有局限或短板
- Opportunities（机会）：哪些场景或趋势让该方案更有价值
- Threats（威胁）：哪些因素可能削弱该方案的适用性（竞品、技术演进等）

**写作步骤**：
1. 先给出方案的一句话定位（它是什么，解决什么核心问题）
2. 依次展开 S/W/O/T 四个维度
3. 在每个维度下给出 2–4 个具体的例子或数据，不要空泛描述
4. 最后给出一个"适合/不适合使用该方案的场景"的总结

**注意**：SWOT 适合帮读者做决策，不适合用来"强行推荐"。应该客观呈现，让读者根据自己的场景判断。

## 权衡三角（Trade-off Triangle）

**适合场景**：讲解一个系统在多个维度之间必须做取舍，且没有"完美方案"时。

**基本思路**：
任何非平凡的系统都在若干关键维度上受到约束，提升一个维度往往意味着在另一个维度上的妥协。最著名的例子是 CAP 定理（一致性 / 可用性 / 分区容忍性，三者只能同时满足两个）。类似的还有：时间/空间/代码可读性，性能/安全/开发成本等。

**写作步骤**：
1. 明确该系统涉及的核心权衡维度（通常 2–3 个）
2. 解释为什么这些维度之间存在内在冲突（根本原因）
3. 给出不同权衡选择下的具体表现（举真实案例或代码示例）
4. 说明在什么场景下应该倾向于哪种权衡

**适用提示**：权衡三角特别适合在讲解系统设计、数据库、网络协议类教程时使用，能帮助读者避免"寻找完美解"的误区。

## 对比学习（Contrast Learning）

**适合场景**：某个概念容易与另一个概念混淆，或读者对"什么时候用 A、什么时候用 B"感到困惑时。

**基本思路**：
通过正例与反例的对比，清晰划定概念的边界。人类对"边界"的感知往往比对"中心定义"更敏锐——看到两个事物的不同之处，比背诵一个定义更容易产生清晰的认知。

**写作步骤**：
1. 给出两个（或多个）容易混淆的概念
2. 先分别给出每个概念的独立定义
3. 构造一个"只有 A 适用"的场景，解释为什么 B 不行
4. 再构造一个"只有 B 适用"的场景，解释为什么 A 不行
5. 最后给出"两者都可以用，但有细微差别"的场景（如果存在）

**常见应用**：
- 进程 vs 线程
- 同步 vs 异步
- REST vs GraphQL
- Docker vs 虚拟机
